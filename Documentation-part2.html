06 — How it works

The job-centric workspace & the engine

The biggest change in this build: the app is organised around the work order as a guided, gated pipeline, not a pile of free tabs. Each step unlocks the next and locks the last — so the process is mistake-proof by design.

Open any W/O and a stepper shows exactly where it stands and the single next action. The steps run in order:

1 · DocumentsSchematics, layout boards (with a 3D view), GA drawings & datasheets — attached before the BOM, visible to every role
2 · BOMThe engine parses the config into feeders and builds the harness. Edit freely, then Finalize & Lock. Locked = read-only; to change it you Revise, which creates Rev n+1 and flags the quote out of date. No silent edits.
3 · Costing & QuoteLocked until the BOM is locked. Material → wastage → labour → overhead → margin → quote, live margin slider, per-unit price. Send the quote (Rev n).
4 · ApprovalThe gate. The quote sits as "Sent" until Approved. After approval, costing locks so the price can't drift.
5 · ProcurementDoes not exist until approval. Required / In stock / On order / Received / To order — order only the shortfall, one click. Receipt updates automatically from Stores.

Why the gates matter

One BOM → one quote → one approval → one set of POs. You can't cost a moving BOM, can't buy before approval, and revising a locked BOM automatically invalidates the quote and voids approval. That's what stops the chaos of multiple costings, quotes and POs drifting out of sync.

The costing engine, concretely

RRL+ME→ parse → R · R · L · ME→ feeder library → harness lines→ material master → priced BOM→ + labour, wastage, OH, margin→ quote

Each feeder type carries a harness set (wires + lengths + hardware) and a build time. The engine aggregates them, prices from the material master, derives labour from total build-minutes, and applies wastage, overhead and your margin. Change the config or quantity and the whole BOM and cost rebuild.

This is the POC proof, running

The feeder library and rates are sensible placeholders — the walkthrough with the owner replaces them with the real wires, lengths and build times. The mechanism is what the POC proves, and it's here.

07 — This build

What changed from your feedback

Every point you raised, addressed.

  • BOM now locks. Review the engine's harness, edit freely, then Finalize & Lock. Locked = read-only; changing it needs a Revise that creates Rev n+1 — no silent edits, no drift.
  • Stage-gated, mistake-proof flow. Costing is locked until the BOM is locked; procurement doesn't exist until the quote is approved. You physically can't raise a PO right after sending a quote — approval comes first.
  • Revising a quoted BOM flags it out of date and voids approval — so one BOM maps to one quote to one approval to one set of POs. No multiplication.
  • A stepper shows the single next action at every point ("Lock the BOM", "Send the quote", "Mark approved", "Raise PO") — the order never feels ambiguous or half-done.
  • Documents moved ahead of the BOM in the flow.
  • Planning restructured around the work order — the cramped modal is gone; each job opens a full workspace with the engine front-and-centre.
  • Costing simplified — the separate "scenarios" table is gone. One margin slider with live total and per-unit price; cost drivers (wastage, labour rate, overhead) are editable but out of the way.
  • "What's to order / what's ordered" is now visible — per job (Procurement tab) and across all jobs (Purchase page), netted against stock and open POs.
  • Projects merged into a Planning dashboard — the old Projects list was just another order table. It's now the planner's command center: live KPIs (to quote, to order, due-this-week, overdue, order book), a "needs your attention" worklist, a pipeline-by-stage view, and the by-client rollup — all clickable straight into the workspace.
  • Progress lives in Orders & BOM — each row shows its stage as an inline progress bar, so you see where every job stands without leaving the list.
  • Purchase is one page, not confusing tabs — "to order by job", then quotes, then the PO register, top to bottom.
  • Receipt is automatic — a PO becomes Partial / Received when Inventory logs the goods inward in Stores. No separate "mark received" step.
  • Costing math is explicit — material, wastage %, labour (build-minutes × rate), overhead %, margin % → quote. Every number is labelled and adjustable.
  • 3D layout board stays — reachable from any "Layout board" document.

09 — Honest review

Where it stands

A straight read for an internal partner conversation.

As a POC it does its job: the costing engine works end-to-end on realistic data, the four modules are connected by one job ID, and the whole shape is visible and credible. The model of the business is right and it speaks the shop's language — W/Os, feeders, GRNs, harnesses.

It is, deliberately, not production software yet: state lives in the browser, logins are demo-only, and there's no shared database — exactly the right trade for a free, no-infrastructure pilot. The point of the POC isn't to ship; it's to prove the engine cheaply and decide together whether to proceed.

The one input that unlocks the rest

The engine is only as good as the owner's harness know-how behind it. The feeder library (wires, lengths, build-times) in this build is a placeholder. The costing walkthrough is the first real input we need — once it's in, we validate against 3–5 past quotes and we have our proof.

What to resist building

A full accounting/GST module (connect Tally instead), HR/payroll, a native mobile app (make the web app responsive first), and deep Gantt scheduling (a due-this-week lane delivers most of the value). Every screen should answer a question someone asks out loud during the working day.

11 — Build & deploy

Architecture & deployment

This POC

  • Static site — no backend, no build step
  • index.html + js/*.jsx (React via in-browser Babel)
  • Costing engine in js/store.jsx; workspace in js/workspace.jsx
  • layout-3d.html — Three.js layout board
  • State in localStorage — swap for an API in Phase 1

Deploy to Vercel

It's static — import the repo (framework: Other, no build), or:

npx vercel          # link / create
npx vercel --prod   # promote

Or drag the folder onto vercel.com. vercel.json enables clean URLs; no env vars needed.

On the Vite build

A vite-app/ production scaffold exists, but the static POC is the up-to-date version we iterate on. We'll regenerate the Vite/TypeScript build from the final design at the start of Phase 1, when the backend goes in — no point compiling a design we're still changing.
